# AI 활용 리포트

본 문서는 PROMPT 기록을 기준으로 AI 활용 방식을 `설계`, `문제 해결`, `생산성` 관점에서 정리한 내용입니다.

## 1. 설계

프로젝트는 백엔드, 프론트, 웹어드민, 앱(Flutter)으로 나누어 진행했습니다.

- 먼저 각 영역에서 요구사항 문서(`PROBLEM.md`, 요구사항 md)를 읽고 기본 프레임워크만 구성했습니다.
- 백엔드는 제가 도메인 구조와 동작을 먼저 정한 뒤 AI에게 구현을 맡겼고, 애매한 부분만 조율했습니다.
- API는 OpenAPI 명세를 먼저 정리하고, 구현이 명세를 따르도록 진행했습니다.
- 프론트/웹어드민은 백엔드 명세를 기준으로 연결 중심으로 설계했습니다.
- 동시성 시나리오 테스트를 먼저 작성하고, 통과 기준으로 구현을 보완했습니다.

## 2. 문제 고민과 해결

진행 중 핵심 이슈는 아래와 같았고, AI를 반복 수정 루프에 활용했습니다.

- 명세-구현 불일치: 헤더(`X-USER-ID`, `X-ROLE`), 응답 형식(`ApiResponse<T>`), DTO 필드 차이를 점검하고 맞췄습니다.
- 동시성/정합성: 룰렛 참여 취소, 유니크 제약, 유저-어드민 동시 작업 시나리오를 테스트 기반으로 검증했습니다.
- 운영 이슈: CORS, 예외 로깅, 시드 데이터, Docker/Render/Neon, CI/CD 오류를 로그 기준으로 수정했습니다.
- 앱(WebView) 이슈: 뒤로가기, 검은 화면, 렌더러 크래시, 네트워크 에러, 로딩/종료 UX를 단계적으로 개선했습니다.
- 개념 확인: `require`, `protected set`, top-level 선언 등은 중간에 확인하여 잘못된 구현을 줄였습니다.

## 3. 생산성 향상

생산성은 `표준 프롬프트 + 병렬 작업 + 테스트 검증`으로 높였습니다.

- 반복 지시를 템플릿화해 초기 세팅 시간을 줄였습니다.
- 영역별로 세션을 분리해 병렬로 작업했습니다.
- 코드는 AI가 초안을 만들고, 저는 검토/수정에 집중했습니다.
- 테스트 실패 시 원인-수정-재검증을 반복해 정확도를 높였습니다.
- 프롬프트 기록 정리를 자동화해 회고와 문서 작성 시간을 줄였습니다.

## 결론

이번 프로젝트에서 AI는 코드 작성 자체보다, 설계 실행 속도와 문제 해결 반복 속도를 높이는 데 가장 효과적이었습니다. 저는 방향과 품질 판단에 집중했고, AI는 구현과 수정 속도를 담당했습니다. 이 방식으로 전체 개발 속도와 품질을 함께 관리할 수 있었습니다.
